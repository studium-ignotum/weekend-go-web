## Build script architecture (proposed)

```
scripts/
  build-html.js      # Reads *.src.html, processes <!-- @include --> markers, writes *.html
partials/
  header.html        # Single canonical nav
  footer.html        # Single canonical footer (with dynamic copyright)
  download-cta.html  # App Store + Google Play CTA with {{APK_VERSION}} placeholder
  icons.svg          # <svg> with <defs><symbol id="icon-star">, <symbol id="icon-faq-chevron">, ...
src/
  index.src.html
  terms.src.html
  privacy.src.html
  community-standards.src.html
```

`scripts/build-html.js` (sketch, ~30 lines):
1. Read `src/*.src.html`
2. Replace `<!-- @include partials/X.html -->` → contents of `partials/X.html`
3. Replace `{{APK_VERSION}}` → value from `package.json` `version` field (or hardcoded for now)
4. Write result to root `*.html`
5. Called from `npm run build` AFTER tailwind compile (so injected partials get tailwind classes recognized)

Wait — Tailwind CLI scans HTML to determine which utilities to compile. If partials live OUTSIDE `index.html` etc., need to update `tailwind.config.js` `content` array to include `partials/**/*.html` and `src/*.src.html`.

- `tailwind.config.js` (`content` includes `partials/**/*.html` + `src/**/*.src.html`)

**DELETE:** none (root HTMLs become build output)

## Risk
- MEDIUM-HIGH — changes build pipeline (`npm run build` chain), changes file structure (`*.src.html` source vs `*.html` output)
- Risk of out-of-sync source/output if build script forgotten
- Mitigation: add `prebuild` hook + check generated files in CI later (Theme 3)
- Risk of broken include: build script must fail loudly nếu missing partial
- A11y/SEO không động vào (head meta giữ riêng per page)

## Acceptance approach
- Build script idempotent: chạy 2 lần ra cùng output
- `diff` giữa pre-refactor vs post-refactor HTML output → chỉ khác ở các chỗ unify (h4 typos đã fixed, copyright dynamic, color tokens unified)
- Browser test: 4 HTML pages render identically (visual smoke test)
